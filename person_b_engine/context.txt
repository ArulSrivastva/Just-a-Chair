## Does Person A's SSE currently support agent-bidding negotiation streaming?

**No — not yet, but the transport itself doesn't need to change.** Here's the precise state:

`broadcast(emergencyId, event, payload)` is generic — it can send *any* event type to connected SSE clients. So technically the pipe is capable of carrying `bid_submitted` events right now. But nothing in the currently-built system actually calls `broadcast()` with that event type. Person A's SSE only fires on:
- `emergency_declared`
- `case_added`
- `resource_status_changed`

There's no `bid_submitted` or `round_completed` event anywhere yet, because the code that would trigger those (the negotiation round itself) hasn't been wired to call `broadcast()` — that logic lives in Person B's and Person C's files, not Person A's.

**So: the gap isn't in Person A's SSE implementation — it's that Person B and Person C haven't hooked into it yet.**

### Exact changes needed

**Person A** — no code changes required. Two small non-blocking items:
1. Document `bid_submitted` and `round_completed` as valid event types clients should expect on the stream (so the frontend person knows what to listen for).
2. Optional sanity check: fire several `broadcast()` calls back-to-back in a throwaway script and confirm a connected SSE client receives all of them, not just the last one. This simulates several bids arriving in quick succession.

**Person B** — change the function signature to accept and call a streaming callback:
```typescript
async function runNegotiationRound(
  cases: Case[],
  resources: Resource[],
  onBid?: (bid: Bid) => void
): Promise<{ bids: Bid[], allocations: Allocation[], explanation: string }>
```
Call `onBid(bid)` the instant *each individual agent's* Groq call resolves — not after `Promise.all` finishes. This is what makes bids appear one-by-one in real time rather than all at once at the end.

**Person C** — wire the callback to the actual broadcast:
```typescript
async function triggerRound(emergencyId: string, roundId: string) {
  const { cases, resources } = await loadState(emergencyId);
  const { bids, allocations, explanation } = await runNegotiationRound(
    cases, resources,
    (bid) => broadcast(emergencyId, 'bid_submitted', bid)
  );
  await saveBids(roundId, bids);
  await saveResult(roundId, allocations, explanation);
  broadcast(emergencyId, 'round_completed', { roundId, allocations });
}
```

   # Frontend Content Spec — Emergency Resource Negotiation Platform

This document lists every page in the product and the exact content to display on
each — headings, copy, labels, empty states, and data fields. It is written so the
frontend person can build against it without needing to read the backend docs.

Realtime pages (marked ⚡) consume the SSE stream at
`GET /emergencies/:id/stream`. Everything else is REST-only.

## 1. Login Page

**Route:** `/login`

- Page title: `Sign in`
- Subtext: `Hospital resource coordination platform`
- Fields: `Email`, `Password`
- Primary button: `Sign in`
- Secondary link: `Forgot password?`
- Error state (bad credentials): `Incorrect email or password. Please try again.`
- Error state (network failure): `Couldn't reach the server. Check your connection and try again.`

*For a 4-hour MVP this page can be skipped entirely with a hardcoded session — if
skipped, note it isn't a design gap, it's a deliberate scope cut.*

## 2. Dashboard (Home) ⚡

**Route:** `/dashboard`

This is the landing page after login — a bird's-eye view of every emergency, active
or resolved.

- Page title: `Emergency Overview`
- Top summary cards (4 across):
  - `Active Emergencies` — count, red accent if > 0
  - `Pending Cases` — count across all active emergencies
  - `Resources Contested` — count of resources with more than one bid in the current round
  - `Avg. Time to Allocation` — rolling average, e.g. `4.2s`
- Primary button, top right: `Declare Emergency`
- List of emergencies, each row showing:
  - Scope badge: `Individual` (blue) or `Mass Casualty` (red)
  - Status badge: `Active` (pulsing dot) or `Resolved` (grey)
  - Department reach: comma list, e.g. `ER, ICU, Surgery`
  - Declared time: relative, e.g. `12 minutes ago`
  - Click target → `View Negotiation`
- Empty state (no emergencies at all): headline `No active emergencies` / subtext
  `Your hospital's resources are fully available. Declared emergencies will appear here.`

## 3. Declare Emergency (Modal or Page)

**Route:** `/emergencies/new`

- Title: `Declare Emergency`
- Field: `Scope` — radio/segmented control: `Individual Patient` / `Mass Casualty Event`
- Field: `Departments Affected` — multi-select: `Emergency`, `ICU`, `Surgery`,
  `Radiology`, `Staffing`, `Equipment`
- Helper text under scope selector:
  - Individual: `A single urgent case. The system will run one negotiation round and resolve immediately.`
  - Mass Casualty: `Multiple cases arriving over time. The system will continuously re-negotiate as new cases and resources change.`
- Primary button: `Declare and Open Negotiation`
- Cancel button: `Cancel`
- Validation error: `Select at least one affected department.`

## 4. Live Negotiation Feed ⚡ — the centerpiece page

**Route:** `/emergencies/:id`

This is the page judges will spend the most time on. It has to visibly tell the
story of agents bidding, scores updating, conflicts resolving, and a final
explainable decision — live, without a refresh.

### 4.1 Header bar
- Emergency scope badge + status badge (same as dashboard)
- `Declared 6 minutes ago` (relative time, live-updating)
- Departments involved, as chips
- Button: `Resolve Emergency` (only visible/enabled once no pending cases remain)

### 4.2 Case Queue panel (left column)
- Section title: `Pending Cases (N)`
- Each case card:
  - Acuity badge, color-coded: 1–2 green (`Low`), 3 yellow (`Moderate`), 4–5 red (`Critical`)
  - Required resource types, as small tags: `ICU Bed`, `Surgeon`, `OR Slot`
  - Status: `Pending` / `Negotiating` / `Allocated`
  - When status flips to `Negotiating`, the card gets a subtle pulsing border —
    this is the visual cue that agents are actively bidding on it right now
- Empty state: `No pending cases. All patients have been allocated resources.`

### 4.3 Live Negotiation Panel (center — the main event)

This is the part that needs the most attention. It should feel like watching a
negotiation happen, not reading a static log.

- Section title: `Negotiation Round #N` with a small `● LIVE` indicator while a
  round is in progress, replaced by `✓ Resolved` once `round_completed` fires
- As each `bid_submitted` event arrives, append a **bid card** to a live-updating
  list (newest at top, or a horizontally scrolling timeline — either works):
  - Resource name + type icon, e.g. `🛏 ICU Bed 4` or `🔪 OR Slot 2`
  - Bid score, shown as a large number and a filled bar, e.g. `87/100`
  - One-line reasoning, directly from the agent's response, e.g.
    `High acuity match, no dependency conflicts`
  - Conditions, if any, as small warning-colored tags, e.g. `⚠ Requires anesthesiologist transfer`
  - Case it's bidding on, as a small reference chip, e.g. `→ Case #C-104`
  - Each new card animates in (slide/fade) rather than appearing instantly —
    this is what makes it *read* as live rather than a static table
- Once `round_completed` arrives, the winning bid(s) get visually highlighted
  (e.g. green border, a `WON` badge) and losing bids for the same case dim
  slightly but stay visible — this is the "explainability" the problem
  statement asks for: judges should be able to see not just what was decided,
  but what was considered and rejected.
- Below the feed, a rolling text line updates per round:
  `Round 4: reprioritized due to new trauma arrival` — this is the "situation
  report" framing discussed for mass events, giving continuity across rounds
  rather than each round feeling disconnected from the last.

### 4.4 Explanation Panel (below or beside the feed)
- Section title: `Why this decision`
- Body: the plain-language explanation string from the round (Mistral-generated),
  rendered as prose, not bullet points — this should read like something a human
  administrator wrote
- Small metadata line under it: `Generated in 1.2s · Round #4 · 3 resources considered`

### 4.5 Resource Status panel (right column)
- Section title: `Resource Availability`
- Grouped by department, each resource shown as a compact row:
  - Status dot: green (`Available`), amber (`Reserved`), red (`Occupied`), grey (`Offline`)
  - Resource label, e.g. `OR Slot 3`
  - If contested this round: small `⚔ Contested` tag
  - Manual override affordance (small `⋯` menu): `Mark Offline`, `Mark Available`
    — this is the human-override path a real admin needs, and doubles as a good
    "look, a human can always step in" talking point for judges

### 4.6 Empty / waiting states
- Before any round has started: `Waiting for negotiation to begin...` with a
  subtle loading pulse
- If a round is taking unusually long (e.g. > 8s): `Still negotiating — this can
  take a few seconds for complex cases.` (prevents it looking frozen/broken)
