# AI Negotiation Engine: Execution & Flow Guide

This document describes the input, execution steps, and output of the Autonomous AI Negotiation Engine for emergency resource allocation.

---

## 1. What the Engine Takes as Input

The orchestrator invokes the engine via the following function:

```typescript
export async function runNegotiationRound(
  cases: Case[],
  resources: Resource[],
  onBid?: (bid: Bid) => void,
  roundId?: string
): Promise<NegotiationResult>;
```

### Input Parameters
*   **`cases` (Array of `Case`):** A list of pending patient emergencies needing resources. Each case contains:
    *   `id` (Unique case ID)
    *   `severity` (Number representing medical priority)
    *   `types` (Array of resource needs, e.g., `["surgery", "icu"]`)
*   **`resources` (Array of `Resource`):** A list of available hospital assets. Each resource contains:
    *   `id` (Unique resource ID)
    *   `type` (`'or_slot' | 'icu_bed' | 'staff' | 'equipment' | 'er_bay'`)
    *   `status` (`'available' | 'occupied' | 'reserved' | 'offline'`)
*   **`onBid` (Callback function, Optional):** A real-time callback handler `(bid: Bid) => void`. Whenever an individual resource finishes bidding, this callback is fired instantly to allow streaming updates over Server-Sent Events (SSE).
*   **`roundId` (String, Optional):** The unique identifier of the current negotiation cycle.

---

## 2. What the Engine Does (Internally)

When invoked, the engine executes a **5-step pipeline**:

### Step 2.1: Local Dependency Validation
Before contacting any AI, the engine screens the resource pool for logical constraints.
*   **Operating Room Dependency:** If a resource type is an `or_slot`, it requires at least one `staff` resource with a status of `available` in the pool.
*   If this dependency is not met, the `or_slot` is excluded immediately to prevent wasteful API calls and invalid allocations.

### Step 2.2: Concurrent Bidding (Llama 3.3 via Groq)
For each valid resource, the engine schedules concurrent API calls:
*   The resource behaves as an autonomous agent.
*   It asks `llama-3.3-70b-versatile` on Groq to evaluate the `cases` queue.
*   It forces a tool call (`submit_bids`), compelling the model to return a structured list of bids with case IDs, scores (priority fit), reasoning, and any conditions.
*   **Instant Streaming:** The moment a resource's bidding call resolves, the `onBid(bid)` callback is triggered to stream the event to the frontend.

### Step 2.3: Greedy Conflict Resolution
Once all bids are returned and combined, the engine runs a deterministic greedy algorithm:
1.  All bids are sorted in descending order of their `bid_score` (highest priority first).
2.  It iterates through the sorted bids list.
3.  For each bid, if both the **case** and the **resource** are still unallocated, it matches them and adds them to the allocation results.
4.  If either has already been allocated in this round, the bid is skipped.

### Step 2.4: Summary Explanation (Mistral Large)
*   The final allocation list and raw bids are passed to `mistral-large-latest`.
*   Mistral is prompted to write a brief, 2-sentence summary explaining *why* these allocations were made.
*   **Resilience Catch:** If Mistral fails or is rate-limited (HTTP 429), the engine catches the error and substitutes a placeholder description, keeping the computed allocations intact.

### Step 2.5: Global Fallback Protection
If any fatal error occurs during the concurrent bidding phase (e.g., API keys missing, complete network downtime), the engine catches the exception and returns a pre-configured safe fallback allocation (assigning the first case to the first resource) to prevent the platform from crashing.

---

## 3. What the Engine Gives as Output

It returns a Promise resolving to a **`NegotiationResult`** object:

```typescript
export interface NegotiationResult {
  bids: Bid[];
  allocations: Allocation[];
  explanation: string;
}
```

*   **`bids` (Array of `Bid`):** The list of all valid bids generated by the LLM agents.
    ```json
    {
      "round_id": "round_123",
      "case_id": "case_99",
      "resource_id": "res_staff_1",
      "bid_score": 85.5,
      "reasoning": "Patient has critical surgery type, staff is available and matching specialized general department.",
      "conditions": []
    }
    ```
*   **`allocations` (Array of `Allocation`):** The final conflict-resolved matches showing which resource goes to which case.
    ```json
    {
      "case_id": "case_99",
      "resource_id": "res_staff_1"
    }
    ```
*   **`explanation` (String):** The 2-sentence natural-language summary generated by Mistral (or the fallback placeholder) explaining the decision logic.
